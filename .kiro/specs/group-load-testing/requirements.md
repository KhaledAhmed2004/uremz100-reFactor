# Requirements Document

## Introduction

This document specifies the requirements for a k6-based load testing suite for the Group module of the Educoin backend. The suite exercises all 18 Group API endpoints under realistic concurrent load using k6 — an industry-standard, open-source external load testing tool. Tests run as a separate OS process against a live server, producing real network latency measurements, a real-time web dashboard, and a shareable HTML report. The suite is designed to be CI/CD-friendly: threshold breaches cause a non-zero exit code that fails the pipeline automatically.

The Group module exposes endpoints under `/api/v1/groups` covering group management (SUPER_ADMIN only), group discovery and membership (all roles), and group content — posts, likes, and comments (members only). Role-based access control is enforced at every endpoint.

## Glossary

- **k6**: Open-source load testing tool (grafana/k6) that runs JavaScript test scripts as an external process against a live HTTP server.
- **VU**: Virtual User — a simulated concurrent user executing the test script in k6.
- **Scenario**: A named k6 execution unit with its own VU count, duration, and executor type.
- **Executor**: k6 scheduling strategy for a scenario (e.g., `constant-vus`, `ramping-vus`, `per-vu-iterations`).
- **Threshold**: A k6 pass/fail assertion on a metric (e.g., `http_req_duration{p(95)} < 1000`). Threshold failure causes k6 to exit with code 99.
- **Fixture**: Pre-seeded test data (users, groups, posts) stored in `load-tests/fixtures.json` by the seed script.
- **Seed Script**: A Node.js script (`load-tests/helpers/seed.js`) that creates test data directly in MongoDB and writes `fixtures.json`.
- **Web Dashboard**: k6's built-in real-time dashboard served at `http://localhost:5665` when `--out web-dashboard` is passed.
- **HTML Report**: A static HTML file generated by `handleSummary` using the `benc-uk/k6-reporter` library, saved to `load-tests/reports/report.html`.
- **SUPER_ADMIN**: Platform administrator role with full access to all group endpoints including create, update, delete, kick member, and pin post.
- **BROTHER**: Regular male user role. Can access BROTHER-typed groups only.
- **SISTER**: Regular female user role. Can access SISTER-typed groups only.
- **JUMMAH**: Additional user role with no group-specific privileges beyond standard member access.
- **Load_Test_Suite**: The complete k6 test suite defined in `load-tests/group.load.js`.
- **Seed_Script**: The Node.js data seeding script at `load-tests/helpers/seed.js`.
- **Web_Dashboard**: The k6 built-in real-time metrics dashboard.
- **HTML_Reporter**: The `benc-uk/k6-reporter` handleSummary integration.
- **Baseline_Scenario**: The single-VU, single-iteration scenario measuring baseline latency per endpoint.
- **Read_Load_Scenario**: The 50-VU concurrent read scenario.
- **Write_Load_Scenario**: The 20-VU concurrent write scenario.
- **User_Journey_Scenario**: The 10-VU full browse→join→post→like→comment flow scenario.
- **Spike_Scenario**: The ramping-vus spike test (5→50→5 VUs).
- **Role_Auth_Scenario**: The 20-VU role-based authorization enforcement scenario.
- **BASE_URL**: The target server URL, defaulting to `http://localhost:3000`.

## Requirements

### Requirement 1: k6 Infrastructure Setup

**User Story:** As a developer, I want a complete k6 load testing infrastructure with seed data, npm scripts, and project structure, so that I can run load tests against the Group API with a single command.

#### Acceptance Criteria

1. THE Load_Test_Suite SHALL be organized in a `load-tests/` directory at the project root containing subdirectories: `config/`, `helpers/`, `scenarios/`, and `reports/`.
2. THE Load_Test_Suite SHALL include a `load-tests/config/thresholds.js` file that exports all threshold constants used across scenarios.
3. THE Load_Test_Suite SHALL include a `load-tests/helpers/auth.js` file that exports token helper functions for obtaining and caching JWT tokens per user role.
4. THE Seed_Script SHALL be located at `load-tests/helpers/seed.js` and SHALL be executable as a standalone Node.js script (not a k6 script).
5. WHEN the Seed_Script is executed, THE Seed_Script SHALL connect to MongoDB using the `MONGODB_URI` environment variable and create the following test fixtures: one SUPER_ADMIN user, at least 10 BROTHER users, at least 10 SISTER users, at least 2 BROTHER-typed groups, at least 2 SISTER-typed groups, and at least 5 posts per group.
6. WHEN the Seed_Script completes successfully, THE Seed_Script SHALL write all created entity IDs and JWT tokens to `load-tests/fixtures.json` so that k6 scenarios can read them at startup.
7. IF the `MONGODB_URI` environment variable is not set, THEN THE Seed_Script SHALL attempt to connect to MongoDB using the undefined value (which will produce a connection error), log a descriptive error message, and exit with a non-zero code.
8. IF the Seed_Script encounters a MongoDB connection error, THEN THE Seed_Script SHALL log a descriptive error message and exit with a non-zero code.
9. THE Load_Test_Suite SHALL include a `load-tests/group.load.js` main entry file that imports all scenario modules and defines the k6 `options` object with all scenarios and thresholds.
10. THE `package.json` SHALL include the following npm scripts:
   - `"load:seed"`: `"node load-tests/helpers/seed.js"`
   - `"load:test"`: `"k6 run --out web-dashboard load-tests/group.load.js"`
   - `"load:report"`: `"k6 run load-tests/group.load.js"`
   - `"load:ci"`: `"k6 run load-tests/group.load.js --out json=load-tests/reports/results.json"`
11. THE `load-tests/reports/` directory SHALL be listed in `.gitignore` so that generated reports are not committed to version control.
12. WHERE the `SKIP_LOAD_TESTS` environment variable is set to `"true"`, THE Load_Test_Suite SHALL exit immediately with code 0 without executing any scenarios.

### Requirement 2: Baseline Performance Measurement

**User Story:** As a developer, I want a baseline scenario that measures the latency of each key Group API endpoint under zero concurrency, so that I have a reference point for identifying regressions.

#### Acceptance Criteria

1. THE Baseline_Scenario SHALL be defined in `load-tests/scenarios/baseline.js` and SHALL use the `per-vu-iterations` executor with 1 VU and 1 iteration.
2. WHEN the Baseline_Scenario executes, THE Baseline_Scenario SHALL send exactly one HTTP request to each of the following 7 endpoints in sequence: `GET /api/v1/groups`, `GET /api/v1/groups/:groupId`, `POST /api/v1/groups/:groupId/join`, `GET /api/v1/groups/:groupId/posts`, `POST /api/v1/groups/:groupId/posts`, `POST /api/v1/groups/posts/:postId/like`, `POST /api/v1/groups/posts/:postId/comments`.
3. WHEN the Baseline_Scenario sends each request, THE Baseline_Scenario SHALL include a valid JWT token for a BROTHER user loaded from `fixtures.json`.
4. WHEN the Baseline_Scenario receives a response, THE Baseline_Scenario SHALL assert that the HTTP status code is in the 2xx range using a k6 `check`.
5. THE Baseline_Scenario SHALL tag each request with a `name` tag matching the endpoint pattern (e.g., `GET /groups`, `POST /groups/:id/join`) so that per-endpoint metrics are visible in the dashboard.
6. THE Baseline_Scenario SHALL apply a threshold of `http_req_duration{scenario:"baseline"} p(95) < 500` milliseconds.
7. WHEN the Baseline_Scenario completes, THE Baseline_Scenario SHALL log the measured p50, p95, and p99 latency for each endpoint to stdout.

### Requirement 3: Concurrent Read Load Test

**User Story:** As a developer, I want a concurrent read load test with 50 virtual users, so that I can verify the Group API handles simultaneous read traffic within acceptable latency bounds.

#### Acceptance Criteria

1. THE Read_Load_Scenario SHALL be defined in `load-tests/scenarios/read-load.js` and SHALL use the `constant-vus` executor with 50 VUs and a duration of 30 seconds.
2. WHEN the Read_Load_Scenario executes, each VU SHALL repeatedly execute the following read operations in a loop: `GET /api/v1/groups`, `GET /api/v1/groups/:groupId`, `GET /api/v1/groups/:groupId/posts`, `GET /api/v1/groups/posts/:postId/comments`.
3. WHEN the Read_Load_Scenario sends requests, THE Read_Load_Scenario SHALL distribute requests across all available BROTHER group fixtures to avoid hotspotting a single group ID.
4. WHEN the Read_Load_Scenario sends each request, THE Read_Load_Scenario SHALL include a valid JWT token for a BROTHER user selected from the fixture pool using round-robin or random selection.
5. THE Read_Load_Scenario SHALL apply the following thresholds:
   - `http_req_duration{scenario:"read_load"} p(95) < 1000` milliseconds
   - `http_req_failed{scenario:"read_load"} rate < 0.01` (error rate below 1%)
6. WHEN a response is received, THE Read_Load_Scenario SHALL assert that the HTTP status code is 200 using a k6 `check`, and SHALL increment a custom counter `read_check_failures` for each failed check.
7. THE Read_Load_Scenario SHALL add a `sleep(1)` pause between iterations to simulate realistic user think time.

### Requirement 4: Concurrent Write Load Test

**User Story:** As a developer, I want a concurrent write load test with 20 virtual users performing join, post, like, and comment operations, so that I can verify the Group API handles simultaneous write traffic without data corruption or excessive latency.

#### Acceptance Criteria

1. THE Write_Load_Scenario SHALL be defined in `load-tests/scenarios/write-load.js` and SHALL use the `constant-vus` executor with 20 VUs and a duration of 30 seconds.
2. WHEN the Write_Load_Scenario executes, each VU SHALL perform the following write operations in sequence per iteration: `POST /api/v1/groups/:groupId/join`, `POST /api/v1/groups/:groupId/posts`, `POST /api/v1/groups/posts/:postId/like`, `POST /api/v1/groups/posts/:postId/comments`.
3. WHEN the Write_Load_Scenario sends a join request, THE Write_Load_Scenario SHALL use a unique BROTHER user per VU (loaded from fixtures) to avoid duplicate-member conflicts.
4. WHEN the Write_Load_Scenario prepares a create-post request, THE Write_Load_Scenario SHALL construct a JSON body with a `content` field containing a non-empty string; the test MAY validate this content preparation without sending the API call if the VU has already reached its iteration limit.
5. WHEN the Write_Load_Scenario sends a comment request, THE Write_Load_Scenario SHALL include a JSON body with a `comment` field containing a non-empty string.
6. THE Write_Load_Scenario SHALL apply the following thresholds:
   - `http_req_duration{scenario:"write_load"} p(95) < 2000` milliseconds
   - `http_req_failed{scenario:"write_load"} rate < 0.05` (error rate below 5%)
7. WHEN a response is received for a write operation, THE Write_Load_Scenario SHALL assert that the HTTP status code is in the 2xx range using a k6 `check`.
8. IF a join request returns HTTP 400 (already a member), THEN THE Write_Load_Scenario SHALL treat this as a non-failure for flow control purposes and continue to the next operation; the k6 status check for that request SHALL still record a failure since 400 is not in the 2xx range.
9. IF a join request returns any status code other than 400 or 2xx (e.g., 500 or 404), THEN THE Write_Load_Scenario SHALL treat this as a test failure and stop the current iteration.
10. THE Write_Load_Scenario SHALL add a `sleep(1)` pause between iterations.

### Requirement 5: User Journey Scenario

**User Story:** As a developer, I want a user journey scenario that simulates 10 virtual users completing a full browse→join→post→like→comment flow, so that I can verify end-to-end latency for the most common user interaction pattern.

#### Acceptance Criteria

1. THE User_Journey_Scenario SHALL be defined in `load-tests/scenarios/user-journey.js` and SHALL use the `constant-vus` executor with 10 VUs and a duration of 30 seconds.
2. WHEN the User_Journey_Scenario executes, each VU SHALL perform the following steps in strict order per iteration:
   - Step 1: `GET /api/v1/groups` — browse available groups
   - Step 2: `GET /api/v1/groups/:groupId` — view a specific group
   - Step 3: `POST /api/v1/groups/:groupId/join` — join the group
   - Step 4: `GET /api/v1/groups/:groupId/posts` — read the group feed
   - Step 5: `POST /api/v1/groups/:groupId/posts` — create a post
   - Step 6: `POST /api/v1/groups/posts/:postId/like` — like the newly created post
   - Step 7: `POST /api/v1/groups/posts/:postId/comments` — comment on the post
3. WHEN the User_Journey_Scenario executes Step 5, THE User_Journey_Scenario SHALL capture the `postId` from the response body and use it in Steps 6 and 7.
4. WHEN the User_Journey_Scenario sends each request, THE User_Journey_Scenario SHALL include a valid JWT token for a unique BROTHER user assigned to that VU from the fixture pool.
5. THE User_Journey_Scenario SHALL apply the following threshold: `http_req_duration{scenario:"user_journey"} p(95) < 3000` milliseconds per step.
6. WHEN a response is received at each step, THE User_Journey_Scenario SHALL assert the HTTP status code is in the 2xx range using a k6 `check` tagged with the step name; IF no response is received at all (network error or timeout), THEN THE User_Journey_Scenario SHALL force the check to fail.
7. THE User_Journey_Scenario SHALL add a `sleep(1)` pause between each step to simulate realistic user interaction time.
8. IF any step returns a non-2xx response, THEN THE User_Journey_Scenario SHALL log the step name, status code, and response body to the k6 console for debugging.

### Requirement 6: Spike Test

**User Story:** As a developer, I want a spike test that ramps VUs from 5 to 50 and back to 5, so that I can verify the Group API recovers gracefully from sudden traffic spikes without sustained error rates.

#### Acceptance Criteria

1. THE Spike_Scenario SHALL be defined in `load-tests/scenarios/spike.js` and SHALL use the `ramping-vus` executor with the following stages:
   - Stage 1: ramp from 0 to 5 VUs over 5 seconds
   - Stage 2: ramp from 5 to 50 VUs over 10 seconds
   - Stage 3: ramp from 50 to 5 VUs over 5 seconds
2. WHEN the Spike_Scenario executes, each VU SHALL perform a mix of read and write operations: `GET /api/v1/groups`, `GET /api/v1/groups/:groupId/posts`, and `POST /api/v1/groups/posts/:postId/like`.
3. WHEN the Spike_Scenario sends requests, THE Spike_Scenario SHALL include valid JWT tokens for BROTHER users loaded from fixtures.
4. THE Spike_Scenario SHALL apply the following thresholds:
   - `http_req_failed{scenario:"spike"} rate < 0.05` (overall error rate below 5%)
   - `http_req_duration{scenario:"spike",stage:"recovery"} p(95) < 1500` milliseconds during the ramp-down stage
5. WHEN the Spike_Scenario completes the ramp-down stage (Stage 3), THE Spike_Scenario SHALL assert that the success rate of requests in that stage is 100% using a k6 `check` tagged `spike_recovery`; IF the ramp-down stage does not complete (e.g., test is interrupted), THE Spike_Scenario SHALL skip this assertion and allow the test to pass based on other thresholds.
6. THE Spike_Scenario SHALL tag requests with a `stage` tag (`ramp_up`, `peak`, `recovery`) corresponding to the current ramping stage so that per-stage metrics are visible in the dashboard.

### Requirement 7: Performance Threshold Assertions

**User Story:** As a developer, I want all performance thresholds defined in a central configuration file and enforced by k6's threshold mechanism, so that CI/CD pipelines automatically fail when performance degrades.

#### Acceptance Criteria

1. THE `load-tests/config/thresholds.js` file SHALL export a `THRESHOLDS` object containing all threshold definitions used by `load-tests/group.load.js`.
2. THE THRESHOLDS object SHALL define the following threshold categories:
   - Read endpoints: `http_req_duration{scenario:"read_load"} p(95) < 1000` and `http_req_failed{scenario:"read_load"} rate < 0.01`
   - Write endpoints: `http_req_duration{scenario:"write_load"} p(95) < 2000` and `http_req_failed{scenario:"write_load"} rate < 0.05`
   - User journey: `http_req_duration{scenario:"user_journey"} p(95) < 3000`
   - Spike test: `http_req_failed{scenario:"spike"} rate < 0.05`
   - Baseline: `http_req_duration{scenario:"baseline"} p(95) < 500`
   - Auth enforcement: `http_req_failed{scenario:"role_auth"} rate < 0.001` (near-zero unexpected failures)
3. WHEN any threshold is breached at the end of a k6 run, THE Load_Test_Suite SHALL exit with a non-zero exit code (k6 exit code 99); IF k6 internally reports conflicting signals, THE Load_Test_Suite SHALL always prioritize the breach signal and exit with code 99.
4. THE `load-tests/group.load.js` options object SHALL import and spread the `THRESHOLDS` object so that all thresholds are active in every run.
5. THE Load_Test_Suite SHALL define per-scenario thresholds using k6's tag-based threshold syntax (`{scenario:"<name>"}`) so that each scenario's thresholds are evaluated independently.
6. WHEN all thresholds pass, THE Load_Test_Suite SHALL exit with code 0.

### Requirement 8: Real-Time Web Dashboard

**User Story:** As a developer, I want a real-time web dashboard available at localhost:5665 during test execution, so that I can monitor VU count, request rate, error rate, and latency percentiles live without any additional tooling.

#### Acceptance Criteria

1. WHEN the `load:test` npm script is executed, THE Load_Test_Suite SHALL pass `--out web-dashboard` to k6, enabling the built-in web dashboard.
2. WHEN the Web_Dashboard is active, THE Web_Dashboard SHALL be accessible at `http://localhost:5665` in a browser during test execution.
3. THE Web_Dashboard SHALL display real-time metrics including: active VU count, HTTP request rate (req/s), HTTP error rate, and `http_req_duration` percentiles (p50, p95, p99).
4. THE Web_Dashboard SHALL display per-scenario metrics when multiple scenarios are running concurrently.
5. THE `load:ci` npm script SHALL NOT include `--out web-dashboard` so that CI runs do not attempt to start the dashboard server.
6. THE `load:report` npm script SHALL NOT include `--out web-dashboard` so that local report-only runs do not start the dashboard server.

### Requirement 9: HTML Report Generation

**User Story:** As a developer, I want an HTML report automatically generated at the end of every test run, so that I can share a self-contained performance report with stakeholders without requiring k6 to be installed.

#### Acceptance Criteria

1. THE `load-tests/group.load.js` file SHALL export a `handleSummary` function that generates an HTML report using the `benc-uk/k6-reporter` library imported from `https://raw.githubusercontent.com/benc-uk/k6-reporter/main/dist/bundle.js`.
2. WHEN a k6 run completes, THE HTML_Reporter SHALL write the HTML report to `load-tests/reports/report.html`.
3. WHEN a k6 run completes, THE HTML_Reporter SHALL also write a human-readable text summary to stdout using `textSummary` imported from `https://jslib.k6.io/k6-summary/0.0.1/index.js`.
4. THE `handleSummary` function SHALL return an object with exactly two keys: `"load-tests/reports/report.html"` mapped to `htmlReport(data)`, and `stdout` mapped to `textSummary(data, { indent: " ", enableColors: true })`.
5. THE generated HTML report SHALL include: overall pass/fail status for each threshold, per-scenario request counts, error rates, and latency percentile tables (p50, p90, p95, p99).
6. IF the `load-tests/reports/` directory does not exist when the report is written, THEN THE HTML_Reporter SHALL create it automatically (k6 creates parent directories for output files).
7. THE `load-tests/reports/` directory SHALL be listed in `.gitignore` to prevent generated reports from being committed.

### Requirement 10: Role-Based Authorization Enforcement Under Load

**User Story:** As a developer, I want a load test that verifies role-based access control is enforced correctly under concurrent load, so that I can confirm authorization middleware does not degrade or fail under traffic.

#### Acceptance Criteria

1. THE Role_Auth_Scenario SHALL be defined in `load-tests/scenarios/role-auth.js` and SHALL use the `constant-vus` executor with 20 VUs and a duration of 10 seconds.
2. WHEN the Role_Auth_Scenario executes, each VU SHALL repeatedly send requests to the following SUPER_ADMIN-only endpoints using a BROTHER user JWT token:
   - `POST /api/v1/groups` (create group)
   - `PATCH /api/v1/groups/:groupId` (update group)
   - `DELETE /api/v1/groups/:groupId` (delete group)
   - `DELETE /api/v1/groups/:groupId/members/:userId` (kick member)
   - `PATCH /api/v1/groups/posts/:postId/pin` (pin post)
3. WHEN the Role_Auth_Scenario receives a response for each unauthorized request, THE Role_Auth_Scenario SHALL assert that the HTTP status code is exactly 403 using a k6 `check`.
4. THE Role_Auth_Scenario SHALL apply the following threshold: `checks{scenario:"role_auth"} rate == 1.0` — meaning 100% of authorization checks must pass (all responses must be 403).
5. WHEN any request in the Role_Auth_Scenario returns a status code other than 403, THE Role_Auth_Scenario SHALL increment a custom counter `auth_bypass_count` and log the endpoint, status code, and response body; THE Role_Auth_Scenario SHALL log these details even if the counter increment operation itself fails.
6. THE Role_Auth_Scenario SHALL also verify that SISTER users receive 403 when attempting to access BROTHER-typed group endpoints that require BROTHER role, asserting HTTP 403 on those responses.
7. THE Role_Auth_Scenario SHALL add a `sleep(0.5)` pause between iterations to maintain a realistic request rate.

### Requirement 11: CI/CD Integration

**User Story:** As a developer, I want the load test suite to integrate with GitHub Actions CI/CD so that threshold failures automatically fail the pipeline, and tests can be skipped in environments where a live server is unavailable.

#### Acceptance Criteria

1. WHERE the `SKIP_LOAD_TESTS` environment variable is set to `"true"`, THE Load_Test_Suite SHALL detect this at startup and exit with code 0 without executing any scenarios or generating reports.
2. WHEN k6 exits with a non-zero code due to threshold failures, THE CI/CD pipeline step running `npm run load:ci` SHALL be marked as failed, blocking downstream deployment steps; the pipeline step SHALL fail on any threshold breach regardless of k6's overall exit status.
3. THE `load:ci` npm script SHALL output results to `load-tests/reports/results.json` using k6's JSON output (`--out json=load-tests/reports/results.json`) for machine-readable CI artifact storage.
4. THE Load_Test_Suite SHALL be executable in a GitHub Actions workflow using the following steps: install k6 via `sudo apt-get install k6`, start the server in the background, wait for the server to be ready, run `npm run load:seed`, then run `npm run load:ci`.
5. THE GitHub Actions workflow step for load testing SHALL set a timeout of at least 5 minutes to accommodate the full test suite duration.
6. WHEN the `load:ci` script is run in CI, THE Load_Test_Suite SHALL NOT start the web dashboard (no `--out web-dashboard` flag) to avoid port binding issues in headless CI environments.
7. THE `load-tests/reports/results.json` file SHALL be uploadable as a GitHub Actions artifact using `actions/upload-artifact` for post-run analysis.
8. IF the server is not reachable at `BASE_URL` when k6 starts, THEN THE Load_Test_Suite SHALL fail immediately with a descriptive connection error rather than running all iterations against a dead server.

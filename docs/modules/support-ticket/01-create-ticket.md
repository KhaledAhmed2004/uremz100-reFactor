# 01. Create Ticket

```http
POST /support-tickets
Content-Type: multipart/form-data
Auth: Bearer {{accessToken}} (BROTHER, SISTER)
```

## 1. Overview
Opens a new support ticket. The endpoint creates the parent `SupportTicket` document AND the first `TicketMessage` (from the requesting user) inside a single Mongoose transaction so the two never diverge.

## 2. Business Rules

### 2.1 Authentication & Roles
- Requires a valid JWT for role `BROTHER` or `SISTER`. `SUPER_ADMIN` is **not** allowed to use this route — admins cannot open tickets on behalf of users.
- `userId` is taken from the JWT (`req.user.id`); it cannot be supplied in the body.

### 2.2 Ticket Number Generation
- The ticket gets a human-readable `ticketNumber` of the form `TCK-####`.
- Generated by an atomic `findOneAndUpdate({ key, $inc: { value: 1 }, upsert: true })` against the internal `supportticketcounters` collection. Two concurrent creates cannot collide on the unique index.

### 2.3 Initial State
- `status` = `OPEN`
- `priority` = body value (or `MEDIUM` default)
- `category` = body value (or `OTHER` default)
- `lastReplyAt` = now
- `lastReplyBy` = `USER`
- `messagesCount` = `1`
- `assignedAdminId` = `null`

### 2.4 File Handling
- Files are received under the multipart field `attachments` (up to 5, ≤ 25 MB each).
- Allowed MIME types follow the global `fileHandler` policy: images (`jpeg/png/jpg/webp`), media (`mp4/webm/audio`), documents (`pdf`).
- After `fileHandler` runs, the upload URLs are exposed at `req.body.attachments` as either a string or string[]. The controller's `buildAttachmentsFromBody` helper maps them into typed `ITicketAttachment[]` (`type` is inferred from extension).

### 2.5 Atomicity
Wrapped in `mongoose.startSession()` / `startTransaction()`:
1. Counter increment for ticket number.
2. `SupportTicket.create([...], { session })`.
3. `TicketMessage.create([...], { session })`.
4. `commitTransaction`.

If any step throws, `abortTransaction` rolls back the ticket so we never leave behind a ticket without its first message. The counter increment is **not** rolled back (counters are intentionally monotonic; the gap is harmless).

### 2.6 Real-time
After commit, `TICKET_CREATED` is emitted to:
- `user::{ownerId}` — for the user's other sessions.
- `admin-tickets` — global broadcast room joined by every connected `SUPER_ADMIN`.

See [11-socket-events.md](./11-socket-events.md).

## 3. Request Body (Form-Data)

| Field | Type | Required | Constraint | Example |
|---|---|---|---|---|
| `subject` | `string` | Yes | 3–200 chars, trimmed | `Payment failed but charged` |
| `message` | `string` | Yes | 5–5000 chars, trimmed | `My card was charged but the subscription didn't activate.` |
| `category` | `enum` | Defaults to `OTHER` | `BILLING`, `ACCOUNT`, `BUG`, `FEATURE`, `OTHER` | `BILLING` |
| `priority` | `enum` | Optional | `LOW`, `MEDIUM`, `HIGH` | `HIGH` |
| `attachments` | `file[]` | Optional | ≤ 5 files, each ≤ 25 MB | — |

### Error: Validation failed (400)
```json
{
  "success": false,
  "statusCode": 400,
  "message": "Validation Error",
  "errorMessages": [
    { "path": "body.subject", "message": "Subject must be at least 3 characters" }
  ]
}
```

### Error: File too large (400)
```json
{
  "success": false,
  "statusCode": 400,
  "message": "File too large for field 'attachments'. Max 25 MB."
}
```

### Error: Forbidden role (403)
*Returned when a `SUPER_ADMIN` token is used.*
```json
{
  "success": false,
  "statusCode": 403,
  "message": "You don't have permission to access this API"
}
```
